Payment Orchestration

Nexus

Payment orchestration layer across multiple providers — normalizing payment data, handling refunds and disputes, and exposing a unified API to the rest of the platform.

Backend BÆRSkin Tactical · DivBrands Production Microservice
◈ Overview

Nexus is a backend payment processing orchestrator that abstracts over multiple payment providers — Stripe, PayPal, Klarna, Primer, and Adyen. It normalizes payment data into a unified model, handles refunds and dispute operations, and exposes a single consistent API that the rest of the platform uses regardless of which provider processed a given transaction.

The project demonstrates deep expertise in payment provider integration, retry logic for financial operations, and the design of reliable orchestration layers in a multi-provider environment.

◈ Features
  • Unified API for receiving payment operation requests from other internal services
  • Normalized payment data model across Stripe, PayPal, Klarna, Primer, and Adyen
  • Asynchronous workers to handle requested operations with retry logic and fallbacks for reliability
  • Refund and dispute operation handling delegated to the correct provider per transaction
◈ Architecture
System Context
Part of a broader microservices ecosystem. Nexus acts as the payment abstraction layer — consumed by services like Resolve and Shield to execute financial operations without needing to know which provider handled the original transaction.
Backend API
Exposes a provider-agnostic API for payment queries and operations. Routes requests to the correct provider integration based on transaction metadata.
Async Workers
Inngest functions execute payment operations asynchronously with retry logic, exponential backoff, and fallback handling to ensure reliability under provider-side failures.
Database
PostgreSQL stores normalized payment records, operation logs, and provider mappings — serving as the source of truth for all payment state across the platform.
